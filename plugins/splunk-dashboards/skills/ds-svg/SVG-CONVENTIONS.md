# SVG conventions for Splunk Dashboard Studio

Shared encoding rules for all SVGs generated by ds-svg. Every SVG
produced by this skill MUST follow these conventions — they ensure
the output works inside Dashboard Studio without manual fixes.

## viewBox sizing

| Mode | viewBox | Rationale |
|---|---|---|
| Icon | `0 0 64 64` | Matches singlevalueicon render size |
| Wordmark / logo | `0 0 400 120` | Standard header logo aspect ratio |
| Background | `0 0 1440 900` | Dashboard canvas size (absolute layout) |
| Rack diagram | `0 0 200 800` | Tall, narrow — 42U rack front view |
| Floor plan | `0 0 1200 800` | Wide, landscape room layout |
| Pipeline | `0 0 1000 200` | Horizontal flow diagram |

Always use `viewBox="0 0 W H"` without fixed `width` / `height`
attributes on the root `<svg>`. The container (Splunk panel) controls
rendered size.

## Self-contained requirements

Every SVG must be fully self-contained. No external dependencies.

- **Fonts:** no external fonts. Convert text to `<path>` elements, or
  use system fonts with a generic fallback:
  `font-family="Helvetica, Arial, sans-serif"`.
- **References:** no `<use>` elements pointing to external files.
- **Images:** no `<image>` elements referencing external URLs.
- **Prologue:** no `<?xml ...?>` declaration.
- **Namespace:** `xmlns="http://www.w3.org/2000/svg"` required on the
  root `<svg>` element.

## Data-URI encoding for inline use

The `splunk.choropleth.svg` inline hack accepts SVGs as data URIs.
Encoding rules:

- `#` in hex colors → `%23` (URL-encode the hash sign).
  `fill="#FF2942"` becomes `fill='%23FF2942'`.
- Use single quotes around attribute values inside the SVG (avoids
  escaping double quotes in JSON):
  `<svg xmlns='http://www.w3.org/2000/svg' ...>`.
- Keep total SVG under ~3 KB for inline use. Larger SVGs should use
  KV-store upload instead.
- No line breaks in the data-URI string — Splunk treats them as
  literal characters and the SVG breaks.
- Format: `data:image/svg+xml;utf8,<svg ...>...</svg>`.

### Encoding example

Source SVG:
```xml
<svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 64 64" fill="none">
  <circle cx="32" cy="32" r="24" stroke="#1a1a1a" stroke-width="4"/>
  <path d="M22 32l7 7 13-13" stroke="#10b981" stroke-width="4" stroke-linecap="round" stroke-linejoin="round"/>
</svg>
```

Data-URI encoded:
```
data:image/svg+xml;utf8,<svg xmlns='http://www.w3.org/2000/svg' viewBox='0 0 64 64' fill='none'><circle cx='32' cy='32' r='24' stroke='%231a1a1a' stroke-width='4'/><path d='M22 32l7 7 13-13' stroke='%2310b981' stroke-width='4' stroke-linecap='round' stroke-linejoin='round'/></svg>
```

## Group ID conventions

Wrap logical sections in `<g>` elements with descriptive `id`
attributes. This makes targeted iteration possible ("make the icon
bigger", "change the wordmark color").

| ID | Use |
|---|---|
| `id="icon"` | Main icon shape |
| `id="wordmark"` | Text portion of a logo |
| `id="background"` | Background layer |
| `id="accent"` | Accent/highlight elements |
| `id="region_name"` | Choropleth regions (must match SPL field values) |

For choropleth canvases, each fillable region must have a unique `id`
that matches exactly the value in the SPL search's `region` field.

## Color handling

### Palette inheritance

When ds-couture has established a design context, use those colors:

- **Primary accent:** the brand or archetype accent color.
- **Outline/stroke:** `#1a1a1a` for light theme, `#e5e5e5` for dark.
- **Fills:** palette colors from ds-ref-color.

### currentColor

Prefer `currentColor` for stroke and fill when the SVG should adapt
to its container's text color. This is especially useful for inline
icons that need to match surrounding text.

### Gradients

Only `<linearGradient>` or `<radialGradient>` defined inline in
`<defs>`. Use palette colors as gradient stops. Never reference
external gradient definitions.

### Two-tone style

The default icon style uses two tones:
- Dark outlines via `stroke` (structural shapes)
- Accent fills via `fill` (emphasis areas, highlights)

This produces icons that read well at small sizes while maintaining
visual interest.

## Small-size legibility (icons)

Icons are rendered at sizes from 64px down to 16px. Rules:

- Minimum `stroke-width`: 2 at 64×64 viewBox (scales to ~0.5px at
  16px — still visible on retina displays).
- Prefer solid fills over thin strokes for shapes that must remain
  visible at 16px.
- No fine details that vanish at small sizes: clock hands, tiny stars,
  thin serifs, small text, intricate patterns.
- Center the icon with ~8px padding: effective content area is 48×48
  within a 64×64 viewBox.
- Mental test: if a detail won't survive 32px rendering, remove it.
