## Test dashboards

Two dashboards cover all six skills:

- `**ds_interactivity_core_dark` / `_light**` — covers
`ds-int-tokens`, `ds-int-inputs`, `ds-int-defaults`, `ds-int-drilldowns`,
`ds-int-visibility`. Five sections (§1–§5), one section per skill, all on
an `absolute` layout.
- `**ds_interactivity_tabs_dark` / `_light**` — covers `ds-int-tabs`. Three
tabs (Overview / Details / Threats), each with its own `grid` layout
in `layoutDefinitions`.

Tabs need a separate dashboard because `layout.type: "absolute"` and
`layout.tabs` are mutually exclusive at the top level.

## QA findings (2026-04-28)

Three rounds of live QA against `ds_interactivity_core` exposed schema
gaps and expression-language portability issues. All findings now
reflected in the live bench + the relevant SKILL.md.


| #   | Finding                                                                                                                                                                                                                                                                      | Skill(s)                          | Status                          |
| --- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------- | ------------------------------- |
| Q1  | `visibility` rejected at panel root with `must NOT have additional properties` — must nest under `containerOptions.visibility`.                                                                                                                                              | ds-int-visibility                     | ✅ fixed                         |
| Q2  | Conditions with `"$tok$" = "lit"` (quoted token) break SPL parser at hyphens with `S0201 Syntax error: "web" at position 5`. Token must be bare: `$tok$ = "lit"`.                                                                                                            | ds-int-visibility                     | ✅ fixed                         |
| Q3  | `isSet($tok$)` rejected by Splunk Enterprise 10.2.x (`S0201 Syntax error: "isSet"`) — Cloud-only function. Use `$tok$ != ""` for portability.                                                                                                                                | ds-int-visibility                     | ✅ fixed (with portability note) |
| Q4  | Visibility panels evaluate against undefined tokens unpredictably on first render. Solved by initialising tokens via `defaults.tokens.default.<name>: { value: "" }`.                                                                                                        | ds-int-defaults, ds-int-visibility        | ✅ fixed                         |
| Q5  | Multiselect token in SPL `IN(...)` produces unquoted comma-joined output (`200,404`); breaks on hyphens/spaces. Use `IN ($tok|s$)` filter.                                                                                                                                   | ds-int-inputs, ds-int-tokens              | ✅ fixed                         |
| Q6  | `drilldown.linkToDashboard.tokens` as `{form.host: $val$}` map silently drops tokens. Use array `[{token, value}]`.                                                                                                                                                          | ds-int-drilldowns                     | ✅ fixed                         |
| Q7  | `viz_token_echo` and `viz_inputs_proof` were `splunk.singlevalue` / `splunk.table` driven by `ds.search` `eval msg=...` — empty-string tokens produced malformed eval. Converted both to `splunk.markdown` with direct `$tok$` interpolation; ds_token_demo dropped as dead. | ds-int-tokens, ds-int-inputs (bench only) | ✅ fixed                         |
| Q8  | `dataSource.name` regex hygiene — bench `name` fields are short and clean, no violations.                                                                                                                                                                                    | (governance)                      | ✅ verified                      |
| Q9  | `viz_inputs_proof` rendered as raw pipe-text — Dashboard Studio's markdown engine does NOT support GFM pipe-tables (despite earlier ds-viz-markdown SKILL claim). Replaced with bullet-list form (`- **Field** — value`). Rewrote `ds-viz-markdown/SKILL.md` to flag tables as unsupported and document that token interpolation IS supported (skill previously said the opposite on both points). | ds-viz-markdown (skill correction), ds-int-tokens, ds-int-inputs (bench) | ✅ fixed |
| Q10 | Free-text input (`search_text`) and dynamic-dropdown (`host_filter`) values containing spaces or special chars broke the rendered SPL — `filter=asfasdfa afasdf` is parser-invalid. Apply `\|s` filter (same as multiselect) → `filter="asfasdfa afasdf"`. | ds-int-tokens, ds-int-inputs | ✅ fixed |
| Q11 | A/B/C visibility panels rendered on different rows because C (markdown without `title`) was sized differently than A (table) and B (singlevalue). Added matching `title` to C and unified all three panel positions (x: 20 / 490 / 960, w: 460, h: 240). | ds-int-visibility (bench layout) | ✅ fixed |
