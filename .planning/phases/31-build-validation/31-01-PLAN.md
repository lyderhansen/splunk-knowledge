Modify the existing formatter checks loop (line 64) and JS checks loop (line 125) to handle Extension API vizs detected by config.json presence. Per D-01, auto-detect per-viz: config.json means Extension mode, formatter.html means Classic mode.

Insert a new section AFTER the formatter checks loop (after line 122) and BEFORE the JS checks loop (line 125). This new section iterates over viz directories, checks for config.json, and if found applies Extension-specific checks.

New Extension API checks section structure:

1. Loop over viz directories: for vizdir in "$APP_DIR"/appserver/static/visualizations/*/
2. Check if config.json exists in vizdir. If not, skip (Classic viz handled by existing loops).
3. If config.json exists, set EXT_FAIL=0 and print header "--- Extension API: $VIZ ---"

Apply these 5 checks (use E## namespace per Claude's discretion from CONTEXT.md):

E01 — config.json valid JSON with required keys:
  Use node -e to parse and check for optionsSchema and editorConfig keys.
  If node unavailable, use grep fallback checking for both strings.
  Grep pattern: grep -q '"optionsSchema"' and grep -q '"editorConfig"'.

E02 — visualization.js uses ESM import, not AMD define:
  Check visualization.js (or src/visualization.js) for ESM import pattern.
  grep -qE 'import.*VisualizationAPI|import.*dashboard-studio-extension' for ESM.
  Also FAIL if grep -q 'define(\[' matches (AMD detected in Extension viz).

E03 — visualization.js has listener-based data access:
  grep -q 'addDataSourcesListener' in the viz JS file.
  FAIL E03 if absent — Extension vizs must use listener pattern, not updateView.

E04 — visualization.js uses columnar data access:
  grep -q 'columns\[' in the viz JS file.
  FAIL E04 if absent — Extension API delivers columnar data, not rows.

E05 — no formatter.html alongside config.json:
  Check that formatter.html does NOT exist in same viz directory.
  FAIL E05 if both config.json and formatter.html coexist — formats are mutually exclusive.

After all 5 checks, print OK if EXT_FAIL=0, set TOTAL_FAIL=1 if any failed.

Also modify the existing formatter checks loop (line 64) to SKIP vizs that have config.json — those are Extension vizs and should not be checked for formatter.html rules. Add a continue guard at the top of the formatter loop body: if config.json exists in the viz dir, skip.

Similarly modify the JS checks loop (line 125) to detect Extension vizs and apply different JS checks. For Extension vizs in the JS loop:
  - Do NOT check for AMD define wrapper (they use ESM)
  - Do NOT run ES5 compliance checks (Extension vizs use modern JS)
  - DO still check for theme detection (grep for addThemeListener OR detectTheme OR getCurrentTheme)
  - DO still check for null guards (safeStr/safeNum/!= null)

Add a format detection variable at the top of the JS loop: IS_EXTENSION=0; if config.json exists in vizdir, set IS_EXTENSION=1. Then wrap AMD/ES5 checks in [ "$IS_EXTENSION" -eq 0 ] guards and add the addThemeListener pattern to the theme detection grep for Extension vizs.

Keep total additions under 60 lines. The existing Classic checks must remain completely unchanged for vizs without config.json.
