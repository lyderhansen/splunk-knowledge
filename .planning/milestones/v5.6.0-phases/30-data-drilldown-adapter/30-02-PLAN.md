Append a new section after the existing ECR-08 content (after line 542) titled "## ECR-09: Extension API Runtime Differences".

Content requirements (per D-02). Cover five sub-patterns, each with a brief explanation and a code snippet or WRONG/RIGHT comparison:

1. **Loading gate** — Data arrives asynchronously via addDataSourcesListener. The callback fires with loading=true before data is ready. MUST check ds.loading and return early before any rendering. Show the guard pattern:
   if (ds.loading) return;

2. **dataSources null check** — ds.dataSources, ds.dataSources.primary, and ds.dataSources.primary.data may each be undefined. MUST use defensive chaining before accessing columns/fields. Show the correct access pattern:
   var primary = ds.dataSources && ds.dataSources.primary;
   var data = primary && primary.data;
   if (!data || !data.columns || data.columns.length === 0) { drawEmptyState(...); return; }
   Note: this replaces the Classic guard (!data || !data.rows || data.rows.length === 0) from ECR-01.

3. **String-to-number conversion** — ALL values in data.columns are strings, even numeric fields. MUST parseFloat or parseInt before any arithmetic, comparison, or Math operation. Show WRONG (direct comparison) vs RIGHT (parseFloat first) pattern. Note that safeNum from ECR-04 already handles this for Classic — Extension code should use the same safeNum helper but be aware that Classic may deliver pre-typed numbers while Extension never does.

4. **iframe sandboxing** — Extension vizs run in an iframe. No access to window.parent, no DOM manipulation outside the viz container (document.getElementById('root')), no window.location of the parent page. Show a WRONG (accessing window.parent.document) vs RIGHT (staying within container) pattern.

5. **No formatData** — Extension API has no formatData lifecycle method. Raw data arrives directly in the listener callback. All field-index resolution and data reshaping that Classic does in formatData must happen in the render function instead. Briefly note that Classic's formatData return value becomes the data argument of updateView; in Extension, the listener callback IS the data arrival point.

End with a "### Rules" subsection (matching ECR-01 through ECR-08 pattern) with 5-6 bullet points summarizing the mandatory checks. Include a note that ECR-09 applies ONLY to Extension API format — Classic vizs continue using ECR-01 through ECR-08 unchanged.

Target ~40 lines total. Follow the established ECR section structure: **Requirement** sentence, code examples with comments, closing Rules list.
