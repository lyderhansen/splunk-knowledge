<decisions>
## Implementation Decisions

### D-01: Hybrid template structure — new files referenced from visualization-js-template.md

Create two new template files in `plugins/splunk-viz-packs/skills/vp-viz/references/`:
- `build-mjs-template.md` — covers EF-01 (IIFE format) and EF-02 (no `external` for the extension package)
- `package-mjs-template.md` — covers EF-03 (bare stanza names in generated visualizations.conf)

`visualization-js-template.md` gains a "Build & Package" section that references both files instead of embedding them. Keeps each template file focused; one extra hop is acceptable.

**Why not all-inline:** visualization-js-template.md would grow from ~300 to ~600+ lines and conflate source-pattern guidance with build/package pipeline guidance.
**Why not full split with no reference:** Claude already reads visualization-js-template.md in vp-viz Step 4 (Extension workflow) — having no pointer there risks skipping the build/package guidance entirely.

### D-02: Verbatim copy from test42 as source of truth

Both new template files contain a verbatim copy of `tests/test42_redbull/redbull_sports_ext/build.mjs` and `package.mjs` with brand-specific values templated:
- `APP_ID` → `{{APP_ID}}`
- Red Bull brand colors (e.g., `accentHex = '#FF0000'`) → `{{ACCENT_HEX}}`
- Brand strings (description, label, author) → `{{...}}` placeholders

Claude during vp-create copies the entire template, fills placeholders, and writes. Pro: zero-fix-first-build guarantee — the proven file ships. Con: longer template files including brand-specific code (CRC32 PNG generation, preview.png logic) — acceptable trade-off because that code is also proven-working and brand-agnostic at runtime.

### D-03: Rewrite misleading "auto-generated by CLI scaffold" note

The current text in visualization-js-template.md line 241 ("Node.js 22.0.0+ required. `build.mjs` and `package.mjs` are auto-generated by the CLI scaffold — do not hand-write them.") is false — there is no CLI scaffold; Claude writes these files. Replace with:

> "Node.js 22.0.0+ required. Claude scaffolds build.mjs and package.mjs during vp-create Step 1 (Extension API) by copying the templates in [build-mjs-template.md](build-mjs-template.md) and [package-mjs-template.md](package-mjs-template.md). Fill `{{APP_ID}}`, `{{ACCENT_HEX}}`, and other placeholders, then write the result to the app root."

### D-04: Plan structure — two waves, Plan 2 depends on Plan 1

**Plan 1 (Wave 1):** Create both new template files (`build-mjs-template.md`, `package-mjs-template.md`) with verbatim content from test42 — this is where EF-01, EF-02, EF-03 land structurally.

**Plan 2 (Wave 2, depends on Plan 1):** Wire references — rewrite the note in `visualization-js-template.md`, add MUST LOAD directives in `vp-viz/SKILL.md` Step 5/6 of the Extension workflow, update `vp-create/SKILL.md` Step 1 (Extension API) to point at the new templates, bump `plugins/splunk-viz-packs/.claude-plugin/plugin.json` version.

Mirrors the Phase 38 success pattern: Wave 1 creates new reference content, Wave 2 condenses/wires existing files.

### D-05: Three findings each have explicit WRONG/RIGHT comparison

Each new template includes a WRONG/RIGHT block at the top calling out the exact fix:
- **build-mjs-template.md**: WRONG `format: 'esm'` / RIGHT `format: 'iife'`; WRONG `external: ['@splunk/dashboard-studio-extension']` / RIGHT no `external` clause.
- **package-mjs-template.md**: WRONG `[${APP_ID}.${vizName}]` / RIGHT `[${vizName}]`.

Mirrors the existing pattern in conf-templates.md and visualization-js-template.md WRONG/RIGHT sections.

### D-06: visualization.js source stays ESM

The IIFE fix lives in the **build config** (esbuild `format: 'iife'`), not the source. `visualization-js-template.md` continues to specify `import { VisualizationAPI } from '@splunk/dashboard-studio-extension'` as the source syntax. Do not change the source ESM examples — they bundle to IIFE at build time.

### Claude's Discretion

- Exact line ordering and section headers inside the two new template files (as long as the WRONG/RIGHT block is near the top)
- Whether to include the full ~250-line package.mjs CRC32 PNG generator or extract it to a shared snippet — recommend keep inline (verbatim per D-02)
- Exact wording of MUST LOAD directives in vp-viz/vp-create SKILL.md (must match style of existing directives like Phase 38's MUST LOAD for ds-ref-jsonata)
- Whether the rewrite of the visualization-js-template.md note keeps the Node.js 22.0.0+ requirement on the same line or splits it

</decisions>
