# Dashboard Composition

**When to load:** Load this file during vp-design step 7 (hand-off), BEFORE writing dashboard JSON.
This reference is the composition layer that sits ON TOP of ds-ref-archetypes and ds-ref-layout-grid.
It applies premium design taste (impeccable / gpt-taste / ui-ux-pro-max principles) to the
absolute-positioned 1920x1080 canvas constraint of Splunk Dashboard Studio. Every generated viz
pack dashboard must pass the composition patterns documented here — branded background, visual
hierarchy, depth through card grouping, and story-first panel arrangement.

---

## Section 1: Purpose and When to Load

This file is loaded once — at the vp-design hand-off, before any dashboard JSON is written.
It gives Claude the composition decisions needed to produce a dashboard that looks like a
professional designer made it, not a default Splunk template. The patterns here are
viz-pack-specific adaptations of the ds-ref-layout-grid / ds-ref-archetypes foundation,
enriched with premium design constraints: asymmetric hierarchy, mood-aware depth, branded
backgrounds, and story-first narrative flow. All layout uses absolute positioning on a
1920x1080 minimum canvas — no responsive, no flex, no relative units.

## Section 2: Background Treatment Menu (DSH-01)

Every dashboard must have a branded background treatment. Choose ONE of three options based on
brand mood. Setting ONLY `backgroundColor` is not enough — it produces an untouched AI signal.

### Option A: Gradient Wash PNG (rich brands)

Generate a 1920x1080 branded gradient PNG at build time using `generate_assets.js` and the
`shared/theme.js` palette. Embed it as `splunk.image` at z=0 (first item in `structure` array),
with `preserveAspectRatio: false` to stretch to fill 1920x1080.

**PNG path pattern:** `/static/app/{app_id}/images/bg_gradient.png`

**Dashboard JSON structure:**
```json
{
  "visualizations": {
    "viz_bg_gradient": {
      "type": "splunk.image",
      "options": {
        "src": "/static/app/{app_id}/images/bg_gradient.png",
        "preserveAspectRatio": false
      }
    }
  },
  "layout": {
    "layoutDefinitions": [
      {
        "id": "layoutDef_main",
        "type": "absolute",
        "options": { "width": 1920, "height": 1080, "backgroundColor": "#0B0E14" },
        "structure": [
          { "item": "viz_bg_gradient", "position": { "x": 0, "y": 0, "w": 1920, "h": 1080 } }
        ]
      }
    ]
  }
}
```

**Why:** Dashboard Studio rectangles are flat-color only — no CSS gradients are possible on
`splunk.rectangle`. A PNG is the ONLY way to achieve a smooth gradient background (D-04 critical
constraint). `backgroundColor` on layout options acts as a fallback color before the PNG loads.

**Note for build:** Gradient PNG is generated by `generate_assets.js` using `dark.bg`,
`dark.accent`, and `dark.panel` from `shared/theme.js`. Requires a Splunk restart after first
app install to serve new static files.

---

### Option B: Flat Tint + Rectangle Layers (minimal brands)

Set `backgroundColor` on layout options for the base tint. Add 1-2 `splunk.rectangle` items
at low `fillOpacity` for zone separation and subtle depth. No generated PNG needed.

**Dashboard JSON structure:**
```json
{
  "visualizations": {
    "viz_bg_wash": {
      "type": "splunk.rectangle",
      "options": {
        "fillColor": "#1e1b4b",
        "fillOpacity": 0.3,
        "strokeColor": "transparent"
      }
    },
    "viz_bg_highlight": {
      "type": "splunk.rectangle",
      "options": {
        "fillColor": "#7c3aed",
        "fillOpacity": 0.04,
        "strokeColor": "transparent"
      }
    }
  },
  "layout": {
    "layoutDefinitions": [
      {
        "id": "layoutDef_main",
        "type": "absolute",
        "options": { "width": 1920, "height": 1080, "backgroundColor": "#0B0E14" },
        "structure": [
          { "item": "viz_bg_wash",      "position": { "x": 0,   "y": 0,   "w": 1920, "h": 1080 } },
          { "item": "viz_bg_highlight", "position": { "x": 400, "y": 0,   "w": 1120, "h": 600  } }
        ]
      }
    ]
  }
}
```

**Rule:** Use BOTH `backgroundColor` AND `splunk.rectangle` together (D-02). `backgroundColor`
sets the base tint; rectangles add zone separation. A lone `backgroundColor` with no rectangles
is a Slop Test failure.

---

### Option C: Radial Accent Glow (medium brands)

Same as Option B but adds a large rounded rectangle or ellipse at low `fillOpacity` (0.03–0.06)
positioned behind the hero zone to approximate a radial glow. Dashboard Studio has no native
radial gradient on rectangles — a large rounded rectangle with high corner radius at low opacity
creates a similar ambient effect.

**Dashboard JSON structure:**
```json
{
  "visualizations": {
    "viz_bg_base": {
      "type": "splunk.rectangle",
      "options": {
        "fillColor": "#0f172a",
        "fillOpacity": 0.5,
        "strokeColor": "transparent"
      }
    },
    "viz_bg_glow": {
      "type": "splunk.rectangle",
      "options": {
        "fillColor": "#6366f1",
        "fillOpacity": 0.05,
        "strokeColor": "transparent",
        "rx": 200
      }
    }
  },
  "layout": {
    "layoutDefinitions": [
      {
        "id": "layoutDef_main",
        "type": "absolute",
        "options": { "width": 1920, "height": 1080, "backgroundColor": "#0B0F1A" },
        "structure": [
          { "item": "viz_bg_base", "position": { "x": 0,    "y": 0,    "w": 1920, "h": 1080 } },
          { "item": "viz_bg_glow", "position": { "x": -200, "y": -200, "w": 1400, "h": 900  } }
        ]
      }
    ]
  }
}
```

Large `rx` (100–300) on a rectangle that extends slightly off-canvas creates a soft ambient
glow from the upper-left / hero zone. `fillOpacity` must stay at 0.03–0.06 to avoid washing
out the panel content.

### Mood-to-Background Mapping

Brand personality drives the choice. This table is guidance, not a hard rule (D-01).

| Mood | Recommended Treatment | Rationale |
|---|---|---|
| Luxury | A (gradient PNG) | Rich jewel tones need smooth gradients |
| Futuristic | A (gradient PNG) | Neon-on-dark needs radial glow energy |
| Speed | A (gradient PNG) | Diagonal energy gradient |
| Playful | A (gradient PNG) | Multi-color gradient |
| Precision | B (flat + rects) | Swiss-watch restraint; clean flat layers |
| Power | B (flat + rects) | Dark dominance, minimal accent |
| Trust | B (flat + rects) | Calm, reliable, no flashiness |
| Organic | B (flat + rects) | Earth tones, muted layers |
| Minimal | B (flat + rects) | Quiet, spacious |

**Light theme note:** For light themes, Option A (gradient PNG) is optional. Default to Option B
with `backgroundColor: "#F8F9FA"` and low-opacity rectangles in a light tinted accent.

## Section 3: Visual Hierarchy (DSH-02)

All patterns use absolute positioning on a 1920x1080 canvas minimum (D-07). Reference the
4 canonical archetypes from ds-ref-archetypes (executive, operational, analytical, SOC/wall)
but adapt for viz pack context: every data panel is a custom Canvas viz, not a built-in
Splunk viz (D-06). This changes sizing — Canvas vizs need more room for their geometry.

**Critical rule (D-05):** No equal-weight grids. No 50/50 splits. No uniform panel sizes.
Every panel grid must read as designed, not generated.

### Golden-Ratio Hero Split (1920px canvas)

- Total usable width (minus 2x20px margins): **1880px**
- Hero panel: **1186px** (0.618 ratio)
- Supporting panel group: **734px** (0.382 ratio)
- Gutter between hero and supporting: **20px**
- Outer margins: **20px** each side

**Asymmetric KPI Strip:**
- Hero KPI: **560px** wide (1.5x supporting)
- Supporting KPIs: **380px** each
- Strip height: 160–180px (with 20px margin = 20–40px from canvas top)

### Zone Layout by Archetype

All y-values assume 1080px canvas height with 20px top/bottom margins.

**Executive / Business Archetype**

```
y=20..200  [KPI Strip — hero KPI 560px + 2x supporting KPIs 380px each]
y=216..580 [Hero viz 1186px wide LEFT + supporting viz stack 734px wide RIGHT]
y=596..760 [Secondary KPI / trend vizes — full width or 3-column split]
y=776..1060 [Detail table or feed — full width]
```

Reading order: KPIs first (is everything healthy?) → hero viz → details.

**Operational / NOC Archetype**

```
y=20..120  [Status strip — service health indicators full width]
y=136..480 [Core metric hero LEFT 1186px + alert feed RIGHT 734px]
y=496..680 [Secondary metrics — 3 equal panels at 600px each]
y=696..1060 [Capacity / queue panels — 2 columns 920px each]
```

Reading order: Status first (what is failing?) → hero metric → details.

**SOC / Security Archetype**

```
y=20..200  [Alert severity KPIs — full width, 4 panels 460px each]
y=216..580 [Geographic / global view hero LEFT 1186px + top threats RIGHT 734px]
y=596..760 [Kill-chain or MITRE stages — full width]
y=776..1060 [Recent events feed — full width]
```

Reading order: Severity first (how bad?) → geographic origin → tactics → events.

**Analytical / Investigation Archetype**

```
y=20..100  [Filter strip — token controls or summary KPIs]
y=116..480 [Correlation viz hero LEFT 1186px + cross-ref KPIs RIGHT 734px]
y=496..700 [Secondary analysis — 2-3 panels]
y=716..1060 [Detail table — full width, sortable, dense]
```

Reading order: Filters → correlation → detail.

### Scale Contrast Rules

- Hero panel must be **1.5x–2x** the size of supporting panels (at minimum)
- Never place three panels of equal size side by side in the primary zone
- KPI hero value: major `fontSize` or paired with sparkline to increase visual weight
- Section labels (if used): 40–48px height for `large` markdown, 24–32px for `default`

## Section 4: Depth via Card Grouping (DSH-03)

Depth is mandatory — panels without shadow cards fail the Slop Test (item 13). Use
`splunk.rectangle` items positioned before panel groups in the `structure` array.

**Z-order rule:** Earlier position in `structure` array = further back in z-order.
Order MUST be: background image/rects first → shadow/glow rects → panels.

**Pitfall:** If shadow rects appear after panels in the array, they occlude the panels.
Every panel group needs its shadow rect placed BEFORE the panel items in the array.

### Mood-to-Depth Mapping (D-08)

| Mood | Depth Treatment | Layers | Card Style |
|---|---|---|---|
| Luxury | Faux glow + glass effect | 3 (canvas + glow + card) | Semi-transparent fill, rx 8 |
| Futuristic | Faux glow, neon accent | 3 (canvas + glow + card) | Sharp cards, accent border |
| Precision | Sharp shadow cards | 2 (canvas + card) | rx 4, tight |
| Power | Heavy dark cards | 2 (canvas + card) | rx 4, dark fill, no glow |
| Speed | Subtle cards | 2 (canvas + card) | rx 4, thin accent border |
| Trust | Soft shadow cards | 2 (canvas + card) | rx 8, gentle shadow |
| Playful | Colorful card backgrounds | 2–3 | rx 12, tinted fills |
| Organic | Soft grouping rects | 2 (canvas + card) | rx 8, warm-tinted |
| Minimal | Minimal or border-only | 1–2 | Border-only or none |

**Layer count by complexity (D-09):**
- Simple packs (3–4 vizs): 2 layers (canvas + shadow cards)
- Complex packs (5–8 vizs): 3 layers (canvas + zone separators + shadow cards)
- Claude decides based on viz count and density

### Shadow Card Recipe (Standard — 2-layer)

Shadow rect surrounds the panel group with 10px inset on all sides.
`fillColor` must be 15–20 HSL lightness units above canvas color to be visible.

```json
{
  "viz_shadow_kpi": {
    "type": "splunk.rectangle",
    "options": {
      "fillColor": "#1A1E28",
      "fillOpacity": 1.0,
      "strokeColor": "transparent",
      "rx": 8
    }
  }
}
```

**Sizing formula:** shadow rect = panel group bounding box + 10px on all sides.

Example — KPI strip at x=40,y=30,w=1840,h=160:
```json
{ "item": "viz_shadow_kpi", "position": { "x": 30, "y": 20, "w": 1860, "h": 180 } }
```

**Minimum lightness difference:** On `#0B0E14` canvas, shadow rect must be at minimum
`#1A1E28` (≈18 lightness units difference). On `#0F1117` canvas, use `#1E2230`. Never
use a shadow rect lighter than the canvas by fewer than 15 HSL lightness units — it will
be invisible on dark displays.

### Faux Glow Recipe (Premium — 3-layer)

Three concentric rectangles with decreasing opacity create a glow halo behind the panel group.

```json
{
  "viz_glow_outer": {
    "type": "splunk.rectangle",
    "options": { "fillColor": "#6366f1", "fillOpacity": 0.04, "strokeColor": "transparent", "rx": 16 }
  },
  "viz_glow_mid": {
    "type": "splunk.rectangle",
    "options": { "fillColor": "#6366f1", "fillOpacity": 0.08, "strokeColor": "transparent", "rx": 12 }
  },
  "viz_card": {
    "type": "splunk.rectangle",
    "options": { "fillColor": "#1A1E28", "fillOpacity": 0.6, "strokeColor": "transparent", "rx": 6 }
  }
}
```

**Sizing — each layer 8px larger than the next on every side:**

```json
{ "item": "viz_glow_outer", "position": { "x": 16, "y": 66, "w": 1888, "h": 208 } },
{ "item": "viz_glow_mid",   "position": { "x": 24, "y": 74, "w": 1872, "h": 192 } },
{ "item": "viz_card",       "position": { "x": 32, "y": 82, "w": 1856, "h": 176 } },
{ "item": "viz_kpi_1",      "position": { "x": 48, "y": 90, "w": 440,  "h": 160 } }
```

**Use faux glow for:** Luxury, Futuristic moods, or when the brand palette has a strong accent
that benefits from the ambient halo effect. Do not use for Minimal or Trust moods.

## Section 5: Narrative Panel Arrangement (DSH-04)

Panel arrangement follows the data narrative (D-10). A dashboard tells a story — the order
panels appear in the visual hierarchy should match how the viewer needs to read the information.
Random grid placement is a Slop Test failure.

### Domain-to-Narrative Mapping

| Domain | Archetype | Narrative Flow | Story Question |
|---|---|---|---|
| SOC / Security | SOC overview | Alert severity → geographic origin → top threats → recent events | "Where are threats and how bad?" |
| Executive / Business | Executive summary | Hero KPI → trend → breakdown → status table | "Is everything healthy?" |
| Operations / NOC | Operational | Service status → core metrics → alert feed → capacity | "What is happening right now?" |
| Analyst / Investigation | Analytical | Filters → correlation → cross-ref KPIs → detail table | "What caused this?" |
| Retail / E-commerce | Executive | Revenue KPI → conversion funnel → category mix → live feed | "How is the business performing?" |
| Healthcare | Operational | Wait time KPIs → capacity gauge → patient flow → queue | "Are we keeping up?" |

Apply this mapping when deciding which panel goes in the hero zone, which go in the supporting
zone, and which go in the detail zone. The first panel the eye lands on should answer the
dashboard's primary question.

### Density Rhythm

Zones flow from sparse to dense as the viewer drills down:

```
TOP (sparse)
  Hero KPI strip — large values, high whitespace
  Hero viz — largest panel, primary message

MIDDLE (medium)
  Primary supporting vizs — standard panel sizes
  Section headers if used (see below)

BOTTOM (dense)
  Detail tables, feeds, smaller panels
  Tight spacing, more data per pixel
```

**Spacing rules:**
- Within a panel group (shadow card): 8px gap between panels
- Between groups / sections: 24–32px gap
- Outer canvas margin: 20px on all sides
- Section header height: 40–48px for `large` fontSize markdown

### Section Headers (Optional — D-11)

Section headers via `splunk.markdown` are OPTIONAL. Use them when:
- 5+ vizs and zone identity would be ambiguous without labels
- SOC / NOC walls where operators scan multiple sections rapidly
- Analytical dashboards where filtering zones need clear demarcation

**Do NOT use them when:**
- 3–4 vizs where zone identity is obvious from sizing
- Executive dashboards that prioritize visual elegance over wayfinding

**Section header JSON:**
```json
"viz_section_header": {
  "type": "splunk.markdown",
  "options": {
    "markdown": "SERVICE HEALTH",
    "fontSize": "large",
    "fontColor": "#A0AEC0",
    "backgroundColor": "transparent"
  }
}
```

**Critical fontFamily constraint:** `splunk.markdown` has a strict enum.
Allowed: `Splunk Platform Sans`, `Splunk Data Sans`, `Splunk Platform Mono`, `Arial`,
`Helvetica`, `Times New Roman`, `Comic Sans MS`. Any other value fails schema validation.
Do NOT use brand fonts in markdown panels — brand fonts are for Canvas vizs only.

**If fontFamily is explicitly set, use only:** `"Splunk Platform Sans"` or `"Arial"`.
If omitted, Dashboard Studio defaults to Splunk Platform Sans.

**Style rules:**
- Executive dashboards: Title Case ("Service Health")
- SOC / NOC walls: UPPERCASE allowed ("SERVICE HEALTH")
- Position: after shadow rects, BEFORE the panel group in the `structure` array
- Height in structure: 40–48px at `large` fontSize

## Section 6: Anti-Patterns

These are Slop Test failures — output that signals AI generation, not design intent.

1. **Equal-weight grid (all panels same size)**
   Every panel the same size communicates "I made a grid." There must be a hero panel 1.5–2x
   supporting panels. No exception. (Slop Test item 2)

2. **No `backgroundColor` set**
   An unset `backgroundColor` leaves the default Splunk canvas color. It is the most visible
   signal that a dashboard was not deliberately designed. Always set explicitly. (Slop Test item 3)

3. **Solid-color rectangle banner**
   A `splunk.rectangle` spanning the full width at the top as a "header bar" reads as
   PowerPoint 2010. Use gradient PNG (Option A), low-opacity zone separator (Option B), or
   no banner at all. (Anti-pattern absolute ban #6)

4. **50/50 symmetric columns**
   Two equal-width columns side by side have no visual hierarchy. Always use 60/40, 65/35,
   or golden-ratio 62/38 splits. (Anti-pattern absolute ban #7)

5. **Uniform spacing everywhere**
   Equal gaps between every element look mechanical. Use 8px within groups, 16px between
   related groups, 24–32px between sections. Let the spacing communicate structure.
   (Anti-pattern absolute ban #8)

6. **Panels without shadow cards**
   Panels floating directly on the canvas background with no depth card look half-finished.
   Either ALL main panels have shadow/glow cards, or NONE do. Inconsistent depth is worse
   than no depth.

## Section 7: Composition Checklist

Run this checklist before finalizing dashboard JSON:

- [ ] `backgroundColor` is explicitly set on layout options (not default)
- [ ] Background treatment matches brand mood (gradient PNG for Luxury/Futuristic/Speed/Playful; flat+rects for others)
- [ ] Hero panel exists at 40–60% canvas width (minimum 900px on 1920px canvas)
- [ ] No equal-size panel grid — hero is at least 1.5x supporting panels
- [ ] Shadow/glow cards are placed BEFORE their panel groups in `structure` array (z-order)
- [ ] All main panel groups have shadow or glow cards (consistent depth)
- [ ] Spacing varies: 8px within groups, 24–32px between sections
- [ ] Panel arrangement follows domain narrative (not random grid)
- [ ] `splunk.markdown` section headers (if used) use only allowed fontFamily enum values
- [ ] No solid-color full-width banner rectangle at top
