# Aurora Design Framework — Design Spec

**Status:** Approved, ready for implementation planning
**Target plugin version:** splunk-dashboards v0.10.0
**Created:** 2026-04-23

Aurora is an extensible design framework for the `splunk-dashboards` plugin that makes generated Splunk Dashboard Studio v2 dashboards look world-class while respecting Splunk's own design language. It codifies four themes, six composition patterns, and a polish scorecard — all native to Dashboard Studio JSON with no CSS, no JavaScript, and no app dependencies.

---

## 1. Goals

1. **Professional aesthetic by default.** A dashboard generated by `ds-create` with no flags should read as polished and intentional, not as a default template.
2. **Four selectable themes** — `pro`, `glass`, `exec`, `noc` — all anchored to Splunk's official palette (splunkui.splunk.com) so output feels native, not a rebrand.
3. **Six composition patterns** that `ds-create` can apply declaratively: `card-kpi`, `hero-kpi`, `sparkline-in-kpi`, `compare-prev`, `annotations`, `section-zones`.
4. **Objective polish scorecard** in `ds-review` — 10 weighted dimensions returning a 0–10 score with per-dimension findings and remediation suggestions.
5. **Splunk Enterprise + Splunk Cloud compatible** — native JSON only, no custom CSS, no JS, no vetted-app requirements.
6. **Extensible by design** — themes, patterns, and templates are registry-based. Adding a new one is a registration call, not a core refactor.

## 2. Explicit non-goals

Aurora v1 will NOT deliver these, and the framework's documentation will say so out loud:

1. **Animations.** Pulsing critical alerts, transitions, keyframes — Dashboard Studio has no animation primitives.
2. **True glassmorphism / backdrop-blur.** The `glass` theme fakes the feeling with radial gradient canvas and stacked low-opacity rectangles. We'll call it "Linear-inspired", not "Linear-identical".
3. **Gradient text on hero KPIs.** Splunk has no gradient-text option. We use a saturated single color against a soft background rectangle instead.
4. **Custom chart fonts.** Only `splunk.markdown` exposes `fontFamily`. Charts inherit from Splunk Web's CSS.
5. **Per-region map overlays that pan/zoom with the map.** Only statically positioned overlays are possible.
6. **Custom CSS or JavaScript.** Required for Cloud compatibility and vetted-app ease of distribution.

## 3. Architecture

### 3.1 Data flow

`ds-create build my-dash --theme glass --pattern hero-kpi,card-kpi` executes:

1. `ds-create` reads `layout.json` + `data-sources.json` and builds the base dashboard JSON (current behaviour, unchanged).
2. Calls `aurora.apply(dashboard, theme="glass", patterns=["hero-kpi", "card-kpi"])`.
3. Aurora engine:
   - Looks up `THEMES["glass"]` → receives tokens, series palette, layout rules.
   - Writes `definition.defaults.visualizations.global.options.backgroundColor` and default series colors.
   - Runs semantic engine (failure/success/latency detection — existing behaviour preserved).
   - For each pattern in the list: looks up `PATTERNS[...]` and calls `pattern.apply(dashboard, tokens)` which mutates the JSON.
4. Returns the completed `dashboard.json`. Downstream `ds-validate` and `ds-deploy` are unchanged.

### 3.2 Extension points

| To add… | Where | How |
|---|---|---|
| New theme (e.g. `cyberpunk`) | `themes.py` | `register_theme(Theme("cyberpunk", tokens=..., series=...))` — no core edits |
| New pattern (e.g. `ticker-strip`) | `patterns/` | `register_pattern(Pattern("ticker-strip", apply=fn))` |
| New flagship template | `templates/` | Drop `<name>.json` in — `ds-template list` auto-discovers |
| New polish dimension | `aurora_score.py` | Add `ScoringRule(name, weight, check=fn)` — score re-normalizes |
| Phase 2 custom viz pack | (reserved) `viz_packs/` | `aurora.register_viz_pack(name)` + Canvas 2D app generated via `/splunk-viz` |

All four registries support directory-scan auto-discovery — drop a Python or JSON file in the right place and it's picked up without core edits.

## 5. Composition patterns

Six patterns. Each is independently applicable, each emits only native Dashboard Studio JSON, each has documented constraints.

### 5.1 card-kpi — depth via layered rectangles

Insert a `splunk.rectangle` before a KPI group in `layout.structure` (array-order is render-order — earlier entries render behind).

```json
{"type":"splunk.rectangle","options":{
  "fillColor":"#15161a","strokeColor":"#2C2C3A",
  "rx":8,"ry":8,"fillOpacity":1
}}
```

**Constraint:** requires `layout.type: "absolute"`. In grid layouts the pattern skips to avoid breaking the grid (engine detects and falls back to per-viz `backgroundColor`).

### 5.2 hero-kpi — oversized signature KPI

Promote one `splunk.singlevalue` to 2–2.5× neighbouring panels. Emit:

```json
{"type":"splunk.singlevalue","options":{
  "majorValueSize":64,
  "sparklineDisplay":"below",
  "trendDisplay":"percent",
  "showSparklineAreaGraph":true,
  "majorColor":"> value | rangeValue(...)"
}}
```

**Constraint:** Requires one panel flagged as `hero: true` in `layout.json` (engine detects or takes the first singlevalue if no flag).

### 5.3 sparkline-in-kpi — mini trendline inside every KPI

For every `splunk.singlevalue` panel whose data source is a time-series (`| timechart` or `| bin _time`), set:

```json
"sparklineDisplay":"below",
"showSparklineAreaGraph":true,
"sparklineStrokeColor":"<theme.accent>",
"sparklineAreaColor":"<theme.accent with alpha 0.15>"
```

Complementary to `hero-kpi`, which already sets sparkline-below on the promoted panel — this pattern applies the same treatment to all remaining singlevalues so the dashboard reads as consistent.

**Constraint:** Only applies to singlevalues backed by a time-series `ds.search`. Engine inspects the SPL and skips non-temporal panels.

### 5.4 compare-prev — dashed previous-period overlay

Rewrites the underlying `ds.search` SPL to include `| timewrap <period>`, then sets per-series styling:

```json
"seriesColorsByField": {"count_latest_day":"#006D9C","count_1day_before":"#6B7C85"},
"lineDashStylesByField": {"count_1day_before":"dashed"}
```

**Constraint:** Only works on `splunk.line` and `splunk.area`. Engine skips for other viz types.

### 5.5 annotations — event markers on time-series

Introduces a secondary `ds.search` producing `_time`, `annotationLabel`, `annotationColor`. Binds it via:

```json
"annotationX":     "> annotationsDs | seriesByName('_time')",
"annotationLabel": "> annotationsDs | seriesByName('annotationLabel')",
"annotationColor": "> annotationsDs | seriesByName('annotationColor')"
```

**Constraint:** Native on `splunk.line`, `splunk.area`, `splunk.column` only. Not available on bar, pie, or table.

### 5.6 section-zones — narrative structure

For dashboards with more than 6 panels. Groups panels into zones labelled by markdown headers (`### Availability`, `### Performance`) sitting inside zone-wide rectangles with `fillOpacity: 0.04` and subtle stroke.

**Constraint:** Requires `absolute` layout. Engine falls back to markdown-only dividers in grid layouts.

### 5.7 Theme → pattern default map

| Theme | Default patterns (overridable with `--pattern`) |
|---|---|
| `pro` | card-kpi + sparkline-in-kpi + compare-prev |
| `glass` | hero-kpi + card-kpi + sparkline-in-kpi |
| `exec` | compare-prev + section-zones + sparkline-in-kpi |
| `noc` | card-kpi + annotations + built-in status-tile |

## 6. Polish scorecard

`ds-review` gains a polish-score section producing `PolishScore(value=0-10, findings=[...])`. Ten dimensions, each weighted:

| # | Dimension | Weight |
|---|---|---|
| 1 | Theme applied (non-default Splunk) | 0.8 |
| 2 | KPI row wrapped in rectangle card | 1.0 |
| 3 | Hero KPI identified and sized correctly | 1.2 |
| 4 | Sparklines on all singlevalues with time-series data | 1.0 |
| 5 | Compare-to-previous-period active on ≥ 1 time chart | 1.2 |
| 6 | Section-zones used when > 6 panels | 0.8 |
| 7 | Max 8 series per chart | 1.0 |
| 8 | Semantic colors used on status KPIs | 1.2 |
| 9 | Whitespace / gutter minimum 20px | 0.8 |
| 10 | Typography consistency (panel titles ≤ 40 chars) | 1.0 |

**Total weight:** 10.0. Each dimension returns pass / partial / fail, multiplied by weight, summed, then normalized to 0–10.

**Output in `review.md`:**

```markdown
## Polish scorecard

**Score: 6.8 / 10**

### Wins
- [pass] KPI row wrapped in rectangle card (+1.0)
- [pass] Semantic colors on status KPIs (+1.2)

### Gaps
- [fail] No compare-to-previous-period — suggest: add `--pattern compare-prev`
- [partial] Only 3 of 5 singlevalues have sparklines
```

## 8. Custom visualization hook (Phase 2, not in Aurora v1)

Sub-plan 10 creates an empty reserved `viz_packs/` directory. In Aurora v2 a contributor can:

1. Generate a custom Canvas 2D viz via `/splunk-viz` (existing skill in the `splunk-forge` marketplace).
2. Drop it into `viz_packs/<name>/` with an `aurora.register_viz_pack(name)` entry.
3. Users invoke via `ds-create --viz-pack <name> --theme noc`.

This enables things Aurora v1 cannot deliver (pulsing critical alerts, gradient text, true animations). Out of scope for v1 because of the token and time cost of Canvas 2D implementation + app-distribution overhead. The architecture does not block it.

## 9. Splunk Enterprise + Cloud compatibility

**Hard requirements met by Aurora v1:**

- No custom CSS.
- No custom JavaScript.
- No app dependencies beyond Splunk's bundled Dashboard Studio.
- All hex values verified against splunkui.splunk.com.
- All JSON options verified against Dashboard Studio v2 reference (Splunk Cloud 10.2, Enterprise 9.4).

**Manual verification step in sub-plan 12:** install `aurora-exec-hero.json` and `aurora-noc-wall.json` flagship templates via `splunk_install_app_local` and confirm they render identically in Enterprise and a Cloud test stack before closing out.
