# v6.0 — Consolidation & Conviction (DRAFT)

**Status:** Draft — not started. Captured 2026-05-23 after Cursor independent reviews (`cursor_review_2026_05_23.md` + `cursor_overkill_2026_05_23.md`).
**Depends on:** v5.8.0 ships (Phases 38-43 complete).

## Why this milestone exists

Independent review (Cursor / Opus 4.7) surfaced two root causes in `splunk-viz-packs`:

1. **Bloat** — 12 MB bundled `node_modules`, 4122 lines of test code shipped, byte-identical script duplication, broken `feedback_*.md` references, version drift between README (4.1.0) and `plugin.json` (5.7.0), "Phase N" / "test NN" development metadata leaked into shipped content.
2. **Architectural incoherence** — 6 layers of design enforcement (Mood × Aesthetic Flavor double-taxonomy, 21 blueprints, DPR/CON/ECR/D-NN/FAIL-code rule families, 100-point design score, 43-item pre-code checklist) that don't compose into a single voice the agent can lean into.

Neither problem is "the plugin is too strict." The plugin is **bloated and self-contradicting**. Strict design discipline is what delivers criteria 3 (no AI slop) — but only when one strong voice applies it consistently, not when 6 overlapping systems argue with each other.

## Project core criteria (re-stated, drives every phase here)

1. **Low friction, high output.** Small prompts → complete viz packs (app + dashboard + many custom vizs).
2. **Zero render errors + secure.** Every generated pack installs cleanly and stays safe.
3. **No AI slop, designer-grade.** Fonts, colors, composition look like a senior UX/UI/graphic designer made them. Every time.

Every phase below must move at least one criterion forward without regressing the others.

## Proposed phases

### Phase A: Hygiene (~1-2 days)
Mechanical cleanup, zero design risk. All items independently verified by Cursor and the orchestrator.

- A-01: De-duplicate `generate_assets.js` (byte-identical across `vp-viz/scripts/` and `vp-create/scripts/`). Move to one canonical location, delegate via wrapper. Apply same pattern to `build_flat.js`.
- A-02: Move `test_*.{js,sh}` files (~4122 lines) out of shipped `vp-viz/scripts/` into `plugins/splunk-viz-packs/tests/`.
- A-03: Decide `node_modules` strategy (12 MB / 27 packages / 1750 files):
  - Option (a): document one-time `npm install --omit=dev --omit=optional` step, `.gitignore` the vendor bundle.
  - Option (b): keep bundled but prune transitives (`undici`, `iconv-lite`, `whatwg-encoding`, `parse5-parser-stream`, `safer-buffer` are likely not strictly needed for a static viz linter).
- A-04: Fix version drift — README and `plugin.json` and the actual implementation counts must agree (README claims "14 blueprints / 10 categories" → actually 21; README claims "4 dimensions in design scoring" → actually 5; vp-init claims 7 questions → actually 8).
- A-05: Eliminate two broken `feedback_*.md` references in shipped content:
  - `vp-viz/SKILL.md:86` references `feedback_light_theme_contrast.md`
  - `vp-viz/references/viz-blueprints.md:55` references `feedback_viz_store_config_fields.md`
  Either inline the content into the plugin docs or remove the dead pointers.
- A-06: Strip "Phase N" / "test NN" / "v5.x" leaks across all skill `.md` files. These are internal development metadata, meaningless to a fresh agent invocation.
- A-07: Fix workflow numbering 3b/3c in `vp-create/SKILL.md` and `vp-design/SKILL.md` — renumber to consecutive integers.

**Outcome:** Plugin loses ~12 MB of bundled deps (per choice), ~4000 lines of test code, ~3000 lines of duplicated scripts, and all dangling internal references.

### Phase B: YAML & trigger routing (~1 day)

- B-01: Audit YAML frontmatter across all 6 skills. Pick one schema based on what the runtime actually honors. Remove no-ops (`disable-model-invocation: false`) and confirm-or-remove non-standard fields (`paths`, `effort`, `arguments`, `argument-hint`).
- B-02: Deconflict trigger phrases — currently "build viz" matches both `vp-viz` and `vp-create`; "themed viz pack" matches both `vp-init` and `vp-design`; "formatter template" / "formatter not working" matches both `vp-viz` and `vp-debug`. Each skill needs non-overlapping trigger sets.
- B-03: Rewrite descriptions in the slightly-pushy Anthropic-recommended style (currently lean dry/descriptive).
- B-04: Fix `vp-init` hand-off — collects 8 fields, forwards 6 (drops app name and data source). Restore the dropped fields.

**Outcome:** The right skill loads first time, every time.

### Phase C: One design voice (~2-3 days — REAL DESIGN WORK)

**Biggest criteria-3 lever in the entire v6.0 milestone.** Collapse 6 overlapping enforcement layers into 1 coherent design vocabulary.

- C-01: Collapse Mood (9 options) × Aesthetic Flavor (10 options) → ONE taxonomy of 5-7 design archetypes. Each archetype owns a specific palette + typography + composition + animation signature.
- C-02: Audit each of the 21 blueprints in `viz-blueprints.md` for which archetype it expresses. Don't cut blueprints (anti-slop tooling) — instead, group them by archetype and sharpen the per-viz guidance to reflect the archetype's voice.
- C-03: Re-purpose `score_design.js` from 5-dimension scoring to hard violation flags only (`flat fill on data`, `accent used as data color`, `no responsive typography`, `archetype contract violation`). Score-as-gate is theater (rewards gradient-stuffing); flag-as-fail isn't.
- C-04: Audit DPR/CON/ECR/D-NN/FAIL-code rule families. Don't flatten (cross-references trace intent→enforcement→debug) but fill the gaps (D02, D04, D06, D07 are missing). Confirm consistent application.
- C-05: Delegate `dashboard-composition.md` to the existing `ds-couture` skill in `splunk-dashboard-studio` instead of duplicating dashboard design across 4 locations.

**Outcome:** When the agent picks an archetype, every downstream decision (palette, type scale, viz selection, composition, animation) flows from that one choice. Designer-grade output by construction, not by post-hoc enforcement.

### Phase D: Extension API as first-class path (~1-2 days)

Currently Extension API is half-shipped: README never mentions it, `(if format=extension...)` branches exist in 13 files, only 2 live test builds (test41, test42_redbull_ext). Either commit fully or pull back.

- D-01: Commit fully. Add Extension API quickstart to README. Ship at least one canonical Extension example pack in `tests/` documented as the reference.
- D-02: Consolidate Extension branching to ONE top-level conditional in `vp-viz` (and one in `vp-create`), not 13 files. Each skill should be format-agnostic in its core flow; Extension diverges at clearly-marked branch points only.
- D-03: Phase 39 (this milestone, v5.8.0) lands the template fixes; v6.0 D-* phases land the user-facing commitment.

**Outcome:** Extension API is no longer "speculative complexity tax."

### Phase E: AST validator strategy (~1 day)

12 MB of bundled deps for ~1440 lines of validation that grep-fallback already covers the load-bearing 80%.

- E-01: Measure what the AST catches that grep doesn't (in real test runs). Tracking required before any cut decision.
- E-02: Make AST opt-in. Plugin ships clean. Users who want strict mode run `npm install` in the vendor dir.
- E-03: OR: drop AST entirely and lean on grep for the 15 high-frequency mistakes. Choose based on E-01 evidence.

**Outcome:** Plugin shrinks dramatically without losing the catch rate that actually matters.

## What v6.0 will NOT do (preserving criteria 2 + 3)

These came up in the review but would HURT criteria 3 (no AI slop) and criteria 2 (no errors) if executed carelessly:

- ❌ Do NOT blindly trim blueprints to 8-10. Blueprints are anti-slop tooling. Cut only if Phase C audit proves specific entries are dead weight.
- ❌ Do NOT inline `edge-cases.md` into the main viz template. The *why* content is exactly the Anthropic-recommended "explain the why" guidance — keep it.
- ❌ Do NOT collapse the 5 rule numbering systems to 1. The cross-references (DPR-03 → D01 → B5) trace design intent through enforcement to debug. Fix gaps, don't flatten.
- ❌ Do NOT gut `score_design.js` wholesale. Re-purpose to hard violation flags (Phase C-03).
- ❌ Do NOT delete `repair_findings.js` without measuring invocation rate. If it's catching real failures, it's earning its place.
- ❌ Do NOT pull back Extension API. Phase 39 makes the templates work; v6.0 D-* commits to it as a user-facing path.
