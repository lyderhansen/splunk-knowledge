
### Key Differences from Classic

| Aspect | Classic (current) | Extension API |
|--------|-------------------|---------------|
| Entry point | `visualization_source.js` | `src/visualization.js` |
| Config UI | `formatter.html` (raw HTML) | `config.json` (declarative JSON) |
| Module format | AMD (`define([...])`) | ESM (`import { ... }`) |
| Build tool | `build_flat.js` or webpack | esbuild (via CLI scaffold) |
| Output bundle | `visualization.js` (AMD) | `visualization.js` (IIFE/ESM) |
| Theme file | `shared/theme.js` (inlined) | Can inline or import |
| Package format | Manual `tar.gz` | `yarn package` -> `.spl` |
| Conf files | Hand-written | Auto-generated by CLI |
| Viz conf key | (none or custom) | `framework_type = studio_visualization` |
| Data format | ROW_MAJOR_OUTPUT_MODE | Columnar (columns[fieldIdx][rowIdx]) |
| Isolation | Shared page context | Sandboxed iframe |

**IMPORTANT:** The deployed app structure (`appserver/static/visualizations/<name>/`) is IDENTICAL between Classic and Extension. Both frameworks look the same on disk in Splunk. The difference is in the runtime behavior and the `framework_type` flag in `visualizations.conf`.

| formatter.html Control | config.json Equivalent |
|------------------------|----------------------|
| `<splunk-text-input>` | `editor.text` |

## 3. Data Binding

### Columnar Data Format

Data arrives via `addDataSourcesListener` or `useDataSources` hook:

```javascript
// Callback shape
function onData({ dataSources, loading }) {
    if (loading) return;
    
    var data = dataSources.primary.data;
    // data.fields = [{name: "host"}, {name: "count"}]
    // data.columns = [["host-1", "host-2", "host-3"], [100, 250, 75]]
    
    var fields = data.fields;   // Array of {name: string}
    var columns = data.columns; // Array of arrays: columns[fieldIndex][rowIndex]
    
    // Access: columns[0][2] = "host-3" (field 0, row 2)
    // Access: columns[1][0] = 100 (field 1, row 0)
}
```

### Critical Data Handling Rules (from best practices)

1. **All values are strings** -- always parseFloat/parseInt for numeric operations
2. **Gate rendering behind loading check** -- data arrives asynchronously
3. **Null-check dataSources?.primary?.data** -- may be undefined
4. **Provide fallback for empty data** -- render "No data" state

### Comparison: Classic vs Extension Data Access

```javascript
// CLASSIC (current) -- ROW_MAJOR
// In formatData():
//   data.rows = [["host-1", 100], ["host-2", 250]]
//   Access: rows[rowIdx][colIdx]

// EXTENSION -- COLUMNAR
// In listener callback:
//   data.columns = [["host-1", "host-2"], [100, 250]]
//   Access: columns[fieldIdx][rowIdx]
//   data.fields = [{name: "host"}, {name: "count"}]
```

### Lifecycle Differences

| Classic | Extension API |
|---------|---------------|
| `formatData(data, config)` | No formatData -- raw data in listener |
| `updateView(data, config)` | Listener callback fires on data change |
| `reflow()` | `addDimensionsListener` callback |
| `remove()` | No explicit remove -- iframe cleanup |
| Synchronous calls | All asynchronous via postMessage |

### Vanilla JS API Pattern (for Canvas 2D vizs)

```javascript
import { VisualizationAPI } from '@splunk/dashboard-studio-extension';

var container = document.getElementById('root');
var canvas = document.createElement('canvas');
container.appendChild(canvas);
var ctx = canvas.getContext('2d');

var state = { data: null, loading: false, options: {}, theme: 'dark', width: 500, height: 300 };

function render() {
    if (state.loading || !state.data) return;
    canvas.width = state.width;
    canvas.height = state.height;
    // ... Canvas 2D drawing code (SAME as Classic) ...
}

VisualizationAPI.addDataSourcesListener(function(ds) {
    state.loading = ds.loading;
    state.data = ds.dataSources && ds.dataSources.primary && ds.dataSources.primary.data;
    render();
}, { invokeImmediately: true });

VisualizationAPI.addOptionsListener(function(opts) {
    state.options = opts.options;
    render();
});

VisualizationAPI.addDimensionsListener(function(dims) {
    state.width = dims.width;
    state.height = dims.height;
    render();
});

VisualizationAPI.addThemeListener(function(t) {
    state.theme = t.theme;  // "dark" or "light"
    render();
});
```

## 4. Theming

### Extension API Theme Access

```javascript
// Vanilla JS
VisualizationAPI.addThemeListener(function(themeState) {
    var isDark = themeState.theme === 'dark';
    // Apply theme to rendering
});

// One-shot read
var currentTheme = VisualizationAPI.getTheme(); // { theme: "dark" | "light" }

// React hook
var { theme } = useTheme(); // "dark" or "light"
```

### Comparison with Classic

| Classic | Extension API |
|---------|---------------|
| `SplunkVisualizationUtils.getCurrentTheme()` | `VisualizationAPI.getTheme()` |
| Returns theme name string | Returns `{ theme: "dark" | "light" }` |
| Called in updateView | Listener-based (reactive) |
| No change notification | `addThemeListener` fires on change |

### Impact on shared/theme.js

Our `shared/theme.js` pattern (DARK/LIGHT token objects with detectTheme()) maps cleanly:

```javascript
// Classic detectTheme():
// var isDark = SplunkVisualizationUtils.getCurrentTheme() === 'dark';

// Extension detectTheme():
// Already reactive via addThemeListener -- store state.theme, 
// select DARK/LIGHT in render() based on state.theme
```

The theme.js token objects (DARK.bg, DARK.text, LIGHT.bg, LIGHT.text, etc.) are UNCHANGED. Only the detection mechanism changes from a synchronous call to a listener pattern.

## 5. Drilldown

### Activation (config.json)

```json
{
    "showDrilldown": true,
    "hasEventHandlers": true
}
```

Both must be true for drilldown to work.

### Two Approaches

**Approach 1: Element-based (addDrilldownListener)**

```javascript
VisualizationAPI.addDrilldownListener(
    domElement,           // HTMLElement to listen for clicks on
    'custom.click',       // Action identifier string
    function() {          // Payload callback -- called on click
        return {
            // Payload shape -- returns DrilldownPayloadState
            // Exact type definition not fully published
        };
    }
);
```

The framework manages the click event internally. When the element is clicked, it calls the payload callback and passes the result to Dashboard Studio's drilldown handlers.

**Approach 2: Programmatic (triggerDrilldown)**

```javascript
VisualizationAPI.triggerDrilldown({
    action: 'custom.click',   // Action identifier
    payload: { ... },          // Drilldown payload
    originalEvent: event       // Optional: original DOM event (for non-click triggers)
});
```

Use triggerDrilldown for: hover-based drilldown, keyboard navigation drilldown, programmatic triggers.

### Token Setting

Enable in config.json: `"canSetTokens": ["dynamic", "static"]`

```javascript
// From best practices docs -- setting tokens via drilldown:
VisualizationAPI.triggerDrilldown({
    action: 'setToken',
    payload: { name: 'selected_host', value: 'host-1' }
});
```

**WARNING:** Token changes propagate dashboard-wide -- all dependent visualizations refresh.

### Drilldown Payload Shape

**LOW CONFIDENCE:** The exact DrilldownPayloadState and DrilldownArgs TypeScript types are not published in the official docs. The API reference describes them in prose:
- DrilldownArgs: object with `action` (string), drilldown payload, and optional `originalEvent`
- DrilldownPayloadState: returned by payloadCallback

Based on the setToken example and Classic API comparison, the likely shape is:
```javascript
{
    action: 'custom.click',    // or 'setToken'
    payload: {
        name: 'fieldName',     // for setToken
        value: 'fieldValue'    // for setToken
    }
}
```

This needs hands-on testing to verify the exact shapes for custom drilldown actions.

### Comparison with Classic Drilldown

| Classic | Extension API |
|---------|---------------|
| `this.drilldown({action: FIELD_VALUE_DRILLDOWN, field: f, value: v})` | `VisualizationAPI.triggerDrilldown({action: '...', payload: {...}})` |
| `data: {field: val}` shape (per memory) | Payload callback or inline payload |
| Sync call in click handler | Async postMessage to parent |

## 8. Canvas 2D Compatibility

**Verdict: FULLY COMPATIBLE**

Extension API vizs run in an iframe with a `<div id="root">` container. You can create Canvas elements, get 2D contexts, and draw exactly as in Classic vizs. The tutorial demonstrates vanilla DOM manipulation (createElement, appendChild, innerHTML) -- Canvas is just another DOM element.

### Canvas Pattern for Extension API

```javascript
import { VisualizationAPI } from '@splunk/dashboard-studio-extension';

// Create canvas
var root = document.getElementById('root');
var canvas = document.createElement('canvas');
root.appendChild(canvas);
var ctx = canvas.getContext('2d');

// State management
var state = {
    data: null,
    loading: false,
    options: {},
    theme: 'dark',
    width: 500,
    height: 300
};

function render() {
    if (state.loading) return;
    if (!state.data || !state.data.columns || state.data.columns.length === 0) {
        ctx.clearRect(0, 0, canvas.width, canvas.height);
        ctx.fillStyle = state.theme === 'dark' ? '#fff' : '#000';
        ctx.fillText('No data', 10, 20);
        return;
    }
    
    canvas.width = state.width;
    canvas.height = state.height;
    
    // Theme selection (same as Classic shared/theme.js)
    var t = state.theme === 'dark' ? DARK : LIGHT;
    
    // Canvas 2D drawing code -- IDENTICAL to Classic
    ctx.fillStyle = t.bg;
    ctx.fillRect(0, 0, canvas.width, canvas.height);
    // ... rest of rendering ...
}

// Wire up listeners
VisualizationAPI.addDataSourcesListener(function(ds) {
    state.loading = ds.loading;
    state.data = ds.dataSources && ds.dataSources.primary && ds.dataSources.primary.data;
    render();
}, { invokeImmediately: true });

VisualizationAPI.addOptionsListener(function(o) { state.options = o.options; render(); });
VisualizationAPI.addThemeListener(function(t) { state.theme = t.theme; render(); });
VisualizationAPI.addDimensionsListener(function(d) { state.width = d.width; state.height = d.height; render(); });
```

### What Changes for Canvas Vizs

| Concern | Classic | Extension |
|---------|---------|-----------|
| Canvas creation | `this.el.appendChild(canvas)` | `document.getElementById('root').appendChild(canvas)` |
| Resize | `reflow()` called by framework | `addDimensionsListener` callback |
| Option read | `config[ns + '.key']` or `this.getOption('key')` | `state.options.key` (no namespace!) |
| Theme detect | `SplunkVisualizationUtils.getCurrentTheme()` | `state.theme` from listener |
| Data access | `data.rows[row][col]` | `data.columns[field][row]` |
| Click handling | `this._onClick` + `drilldown()` | `addDrilldownListener` or `triggerDrilldown` |
| Animation | `requestAnimationFrame` | `requestAnimationFrame` (same) |

### What Does NOT Change

- All Canvas 2D drawing code (fillRect, arc, fillText, etc.)
- shared/theme.js token objects (DARK/LIGHT)
- Color utility functions (hexFromSplunk, withAlpha, lerpColor, etc.)
- Text measurement and layout logic
- Hit-test geometry for click handling
- Animation frame loops
- Glow/shadow rendering techniques

## 10. Best Practices (from official docs)

### Critical Rules

1. **Always provide fallback values** -- options may be unset regardless of optionsSchema defaults
   ```javascript
   var color = state.options.barColor ?? '#4e9cf5';
   ```

2. **All data values are strings** -- explicit parsing required
   ```javascript
   var values = columns[1].map(function(v) { return parseFloat(v); });
   ```

3. **Gate rendering behind loading** -- never render before data arrives

4. **setOptions is silently ignored in view mode** -- use addModeListener to show/hide edit affordances

5. **No window.parent access** -- sandboxed iframe, all communication via postMessage

6. **Token changes are dashboard-wide** -- setting a token refreshes all dependent vizs

### Anti-Patterns

- Rendering before data arrives (causes errors or blank displays)
- Treating string values as numbers without conversion
- Attempting to modify options in view mode
- Trying to access parent window or Dashboard Studio DOM
- Ignoring loading state from data source listener
