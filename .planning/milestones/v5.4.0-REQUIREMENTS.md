# Requirements: splunk-knowledge — v5.4.0 Runtime Robustness & Visual Polish

**Defined:** 2026-05-19
**Core Value:** When a user runs `/vp-init`, the resulting viz pack installs in Splunk without errors and produces a dashboard that makes someone say "wait, that's Splunk?" — zero manual fixes, wow-factor by default.

**Reference apps studied:** DataDrivers F1 (14 vizs), network-diagram-viz — both use search_fragment + formatData() field indexing, zero data-field formatter controls on most vizs.

## v1 Requirements

### Data Binding (DB)

- [x] **DB-01**: Generated vizs use `formatData()` field indexing pattern — build a field-name-to-column-index map from `data.fields`, access columns by name, not position. No hardcoded field names in JS fallbacks.
- [x] **DB-02**: Formatter has NO "Data configurations" section with field-name text inputs for most viz types. Field binding is implicit via `search_fragment` + `formatData()`. Only generic vizs (single-value, generic table) may have a single optional `field` input with empty default.
- [x] **DB-03**: `search_fragment` in visualizations.conf contains the expected SPL shape for each viz (e.g., `| table rank, name, score, delta`). This is the data contract — users know what columns to provide.
- [ ] **DB-04**: Formatter section labels use generic `{{VIZ_NAMESPACE}}` naming — no domain-baked strings like "Artist Leaderboard: Data configurations"

### Drilldown Wiring (DR)

- [x] **DR-01**: Every clickable viz in a generated pack has `_onClick` implemented in JS with hit-test logic and `this.drilldown()` call
- [x] **DR-02**: Every viz panel with `_onClick` has matching dashboard JSON: `"drilldown": "all"` in options AND `eventHandlers` with `drilldown.setToken`
- [x] **DR-03**: `seriesColors` and `seriesColorsByField` options documented in dashboard-interactivity.md for built-in viz panels (splunk.area, splunk.line, etc.) in mixed dashboards

### Validator Fixes (VF)

- [x] **VF-01**: validate_dash.js DS4 check accepts both `item.item` and `item.vizId` — no false positives on valid dashboard JSON
- [x] **VF-02**: D11 validator scans from `_onMouseMove` method definition position, not first string occurrence — no comment hacks needed
- [x] **VF-03**: validate_dash.js DS5 check verifies wildcard default value `"*"` specifically, not just presence of a `defaults.tokens.default` entry

### Animation Implementation (AN)

- [x] **AN-01**: `showEntrance` formatter toggle triggers a visible entrance animation (rAF arc fill, row cascade, or opacity fade) — toggling Off stops it immediately
- [x] **AN-02**: `flashCritical` formatter toggle triggers visible LED pulse on critical-status elements — shadowBlur oscillation at 500-800ms
- [x] **AN-03**: `animationSpeed` formatter control produces measurably different timing — "slow" takes longer than "fast"

### Visual Polish (VP)

- [x] **VP-01**: Every viz in a pack with `fillTechnique: gradient` in the visual language has at least one `createLinearGradient` or `createRadialGradient` call
- [x] **VP-02**: Light theme renders with WCAG AA contrast (4.5:1 minimum on hero text) — verified by switching `themeMode` to light
- [x] **VP-03**: `seriesColors` JSON array option documented and used for built-in viz panels in generated dashboards — brand palette applied to splunk.area, splunk.line, etc.

## Future Requirements

- Aviation and Fintech domain-specific viz types
- score_design.js automated aesthetic scoring
- Auto-palette HSL derivation for series colors
- Real Splunk data integration patterns (production SPL templates)
- Token-based drilldown configuration in formatter (like network-diagram-viz Tokens section)

## Out of Scope

- DOS expressions (`> x | getField()`) for custom viz field binding — platform feature only available to built-in vizs
- Splunk-native palette dropdown grid in custom viz formatters — platform UI component, not available to custom formatter HTML
- Responsive dashboard layouts — Dashboard Studio uses absolute positioning only
- Real-time streaming vizs — standard Splunk polling is sufficient
