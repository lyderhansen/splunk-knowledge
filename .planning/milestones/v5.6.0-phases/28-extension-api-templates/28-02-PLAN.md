**Section 1: Header and purpose**
- Title: "visualization.js Template (Extension API)"

**Section 2: Complete visualization.js template**
- Full ESM module template in a single JavaScript code fence. This is the primary deliverable. Structure:

  a) ESM import: import VisualizationAPI from '@splunk/dashboard-studio-extension' (per D-04)
  b) CSS import: import './visualization.css' with a comment noting it is optional for Canvas vizs
  c) Inlined theme tokens (per D-04): var DARK = { {{DARK_TOKENS}} }; var LIGHT = { {{LIGHT_TOKENS}} }; — comment referencing theme-template.md for the full token object shape. Include VISUAL_LANG placeholder too.
  d) Utility functions: safeStr, safeNum, hexFromSplunk — identical to Classic (these are plain JS, not framework-dependent). Also include withAlpha and lerpColor from theme-template.md since they are inlined.
  e) Canvas setup: var root = document.getElementById('root'), create canvas element, append to root, get 2d context. Set canvas.style.width and height to '100%'.
  f) State object: var state = { data: null, loading: false, options: {}, theme: 'dark', width: 500, height: 300 }
  g) render() function with:
     - Loading gate: if (state.loading) return
     - No-data fallback: if data is null or columns is empty or columns[0].length is 0, clear canvas, draw "No data" in brand font using theme text color — not a blank panel, not a JS error
     - Set canvas.width and canvas.height from state dimensions
     - Theme selection: var t = state.theme === 'dark' ? DARK : LIGHT, then themeMode override from state.options.themeMode
     - Columnar data access: var fields = data.fields (array of name objects), var columns = data.columns (array of arrays), var rowCount = columns[0].length
     - Option reads: var opt = state.options; then bare property access like opt.barColor or opt.showGlow with nullish coalescing fallback (opt.barColor ?? t.accent). Note: parseFloat for numeric options since all config values arrive as strings.
     - {{CANVAS_RENDER_CODE}} placeholder where the viz-specific drawing code goes
  h) Listener wiring — four listeners:
     - addDataSourcesListener with invokeImmediately: true. Extract ds.dataSources.primary.data with null-safe chain. Set state.loading and state.data. Call render().
     - addOptionsListener: set state.options = o.options, call render()
     - addThemeListener: set state.theme = t.theme, call render()
     - addDimensionsListener: set state.width = d.width, state.height = d.height, call render()
  i) Drilldown wiring (optional section, controlled by config.json showDrilldown): addDrilldownListener on canvas with 'custom.click' action and payload callback that returns row data. Comment explaining the alternative triggerDrilldown approach for programmatic drilldown.

**Section 3: Classic vs Extension lifecycle comparison**
- Side-by-side table: Classic method -> Extension equivalent.
  formatData -> no equivalent (raw data in listener), updateView -> render() called from listeners, reflow -> addDimensionsListener, getInitialDataParams -> config.json dataContract, this.drilldown() -> addDrilldownListener/triggerDrilldown, SplunkVisualizationUtils.getCurrentTheme() -> addThemeListener, config[ns + '.key'] -> state.options.key (no namespace).

**Section 6: WRONG/RIGHT patterns**
- WRONG: Using AMD define() syntax. RIGHT: ESM import { VisualizationAPI }.
- WRONG: Accessing data.rows[row][col] (Classic row-major). RIGHT: data.columns[fieldIdx][rowIdx] (Extension columnar).
- WRONG: Reading options with namespace prefix (state.options['myapp.myviz.barColor']). RIGHT: Bare name (state.options.barColor).
- WRONG: Using this.el or SplunkVisualizationBase. RIGHT: document.getElementById('root').
- WRONG: Treating column values as numbers without parseFloat. RIGHT: Always parseFloat/parseInt since all values arrive as strings.
- WRONG: Rendering before checking loading state. RIGHT: if (state.loading) return as first line of render().
