Create the file with these sections, following formatter-patterns.md conventions (exact copy-paste templates, fill-placeholder style, WRONG/RIGHT comparisons where applicable):

**Section 1: Header and purpose**
- Title: "config.json Template (Extension API)"
- Brief statement: this is the Extension API equivalent of formatter.html — declarative JSON replaces raw HTML. Copy the template and fill only the marked placeholders.
- Note that editor.select and editor.radio are NOT confirmed in Splunk 10.4 docs; use editor.text as fallback for dropdowns.

**Section 2: Complete config.json template**
- Full JSON template using descriptive fill placeholders matching the project pattern: use double-brace descriptive names like {{VIZ_LABEL}}, {{VIZ_DESCRIPTION}}, {{INITIAL_WIDTH}}, {{INITIAL_HEIGHT}}, {{OPTIONS_SCHEMA_ENTRIES}}, {{EDITOR_CONFIG_SECTIONS}}.
- Top-level properties: showTitleAndDescription (true), includeInToolbar (true), includeInVizSwitcher (true), showDrilldown (false by default), hasEventHandlers (false by default), canSetTokens ([]).
- config.name = {{VIZ_LABEL}}, config.description = {{VIZ_DESCRIPTION}}, config.category = "Custom", config.dataContract.requiredDataSources = ["primary"], config.size with {{INITIAL_WIDTH}} and {{INITIAL_HEIGHT}}.
- optionsSchema with themeMode (type string, default "auto") as always-present entry plus {{OPTIONS_SCHEMA_ENTRIES}} placeholder.
- editorConfig array with {{EDITOR_CONFIG_SECTIONS}} placeholder.

**Section 3: optionsSchema type reference**
- Table mapping JSON types to editor types: string -> editor.text or editor.color; number -> editor.number or editor.text; boolean -> editor.checkbox.
- Show the exact JSON shape for each type with default values.

**Section 4: Editor type templates (one subsection per type)**
- editor.color: JSON snippet for a color option. Label, option name binding, type string in optionsSchema.
- editor.text: JSON snippet for text input. Label, option name binding, type string or number in optionsSchema.
- editor.number: JSON snippet for numeric input. Label, option name binding, type number in optionsSchema.
- editor.checkbox: JSON snippet for boolean toggle. Label, option name binding, type boolean in optionsSchema.
- Each shows both the optionsSchema entry AND the editorConfig layout entry as a matched pair.

**Section 5: Section grouping**
- How to group multiple editors under one editorConfig section label (the layout array-of-arrays pattern from the research). Show a complete 3-section example: "Theme", "Data configurations", "Effects" — mirroring the section structure Claude already uses for Classic formatters.

**Section 6: Drilldown wiring**
- When a viz supports drilldown, set showDrilldown: true, hasEventHandlers: true, canSetTokens: ["dynamic"].
- Show the exact top-level JSON diff between a non-drilldown and drilldown-enabled config.json.

**Section 7: Comparison table — Classic vs Extension**
- Two-column table mapping formatter.html controls to config.json equivalents:
  splunk-text-input -> editor.text,
  splunk-color-picker -> editor.color,
  splunk-radio-input -> editor.text (radio not confirmed),
  splunk-select-input -> editor.text (select not confirmed),
  form + section-label -> editorConfig label sections,
  name={{VIZ_NAMESPACE}}.key -> "option": "key" (no namespace!).
- Call out the critical improvement: config.json uses bare property names — the {{VIZ_NAMESPACE}} footgun from Classic is eliminated.

**Section 8: WRONG/RIGHT patterns**
- WRONG: Using namespaced option names ("myapp.myviz.barColor") in config.json option field.
  RIGHT: Bare names ("barColor").
- WRONG: Omitting optionsSchema entry for an editor control (control appears but value is never stored).
  RIGHT: Every editorConfig option has a matching optionsSchema entry.
- WRONG: Setting showDrilldown without hasEventHandlers (drilldown UI appears but clicks do nothing).
  RIGHT: Both showDrilldown and hasEventHandlers set together.

Target length: 150-200 lines. All content in English. Use JSON code fences for templates.
